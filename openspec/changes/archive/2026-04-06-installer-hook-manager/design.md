## Context

Change 1 (project-scaffolding-core-utilities) established the foundational modules: `meta.ts` for `.pulse-meta.json` persistence, `platform.ts` for OS-specific hook command generation, `skills-registry.ts` for skill definitions, and `logger.ts` for output. The installer and hook-manager modules sit on top of these foundations to perform the actual file system operations that `pulse init`, `pulse sync`, and `pulse uninstall` need.

Currently there is no mechanism to write SKILL.md files to `.claude/skills/` or to manage the SessionStart hook in `.claude/settings.json`. These two modules close that gap.

## Goals / Non-Goals

**Goals:**
- Safely write pulse-managed skills to `.claude/skills/<skill-id>/SKILL.md` with `_pulse: true` frontmatter
- Provide ownership detection via `_pulse: true` marker so pulse never overwrites user-created skills
- Merge the SessionStart hook into `.claude/settings.json` without corrupting existing user hooks
- Support idempotent add/remove for both skills and hooks

**Non-Goals:**
- Fetching or transforming skill content (handled by fetcher.ts and transformer.ts)
- Managing hooks other than SessionStart
- Validating the full `.claude/settings.json` schema beyond the hooks section
- Supporting skill installation outside the project directory (global/user-level)

## Decisions

### D1: Frontmatter `_pulse: true` as ownership marker

Skills written by pulse include `_pulse: true` in their YAML frontmatter. The `isManaged()` function parses only the frontmatter block (between `---` delimiters) and checks for this key. This avoids needing an external registry to track ownership and makes detection self-contained per file.

**Alternative considered:** Tracking managed skills only in `.pulse-meta.json`. Rejected because a user could delete the meta file and pulse would lose track, risking overwrites of user skills.

### D2: JSON-level merge for settings.json

`hook-manager.ts` reads the entire `.claude/settings.json`, parses it, modifies only `hooks.SessionStart`, and writes back. It preserves all other keys and formatting by using `JSON.stringify(obj, null, 2)`. The `_pulse: true` property on the hook entry serves as the marker for identification during removal.

**Alternative considered:** Regex-based insertion. Rejected because JSON manipulation is safer and handles edge cases (trailing commas, nested structures).

### D3: Installer writes to `.claude/skills/` relative to `process.cwd()`

The installer always operates project-level. The base path is `path.join(process.cwd(), '.claude', 'skills')`. This matches Claude Code's project-level skills resolution.

### D4: `install()` accepts pre-rendered content

The `install(skills)` function receives an array of `{ id, content }` objects where `content` is the final SKILL.md body (already transformed). The installer is not responsible for transformation — it prepends the `_pulse: true` frontmatter and writes the file.

### D5: Atomic-ish writes

Both modules use `fs.mkdirSync` with `{ recursive: true }` for directory creation and `fs.writeFileSync` for file writes. No tmp-file-then-rename pattern to keep complexity low. The risk of partial writes is acceptable given these are local project files easily regenerated by `pulse sync --force`.

## Risks / Trade-offs

- **[Concurrent writes to settings.json]** → If another tool writes to `.claude/settings.json` simultaneously, data could be lost. Mitigation: this is a CLI tool run manually or via hook — concurrent writes are unlikely.
- **[Frontmatter parsing brittleness]** → If a user edits a pulse skill's frontmatter and removes `_pulse: true`, pulse will treat it as a user skill and skip it. This is by design — the user has claimed ownership.
- **[JSON formatting loss]** → Re-serializing settings.json with `JSON.stringify(obj, null, 2)` may change formatting from the original. Mitigation: this is the standard Claude Code formatting for settings.json.
